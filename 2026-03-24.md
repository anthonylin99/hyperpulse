## Poke holes (potential issues before wider release)

### High-priority

1. **Blueprint mismatch: testnet requirement vs current mainnet flag**
- Current code has `IS_TESTNET = false` in `src/lib/hyperliquid.ts`.
- Original blueprint explicitly required testnet-only until approval to switch.
- Risk: users may assume safe testnet behavior but place real trades.

2. **Private key stored in `sessionStorage` (MVP-acceptable, still risky)**
- Any successful XSS on the app can read the API key from session storage.
- Risk severity is constrained by Hyperliquid agent-wallet permissions, but still meaningful.

3. **Connect success does not guarantee trade permission on API wallet**
- Current connect verification checks `openOrders` + `clearinghouseState` (main wallet visibility).
- It does not proactively place or dry-run an authenticated trade capability check.
- Possible failure mode: user “connects successfully” but first order fails due to API wallet authorization/config mismatch.

### Medium-priority

4. **No retry/backoff strategy in wallet verification path**
- Temporary network/API hiccups can cause hard connect failures.
- User impact: flaky connect UX even with valid credentials.

5. **Error messages may expose low-level upstream details**
- Surfacing raw errors is useful for debugging, but may include confusing or noisy SDK transport text.
- Consider mapping known errors to cleaner user-facing copy and preserving raw details in console only.

6. **No “show/hide key” UX or copy safety guardrails**
- Key field is masked, but there are no helper affordances (visibility toggle, paste cleanup hints, accidental whitespace detection messaging).
- Not a blocker, but creates avoidable support friction.

### Lower-priority

7. **No telemetry around connect failures**
- No structured metrics for failure reasons by type.
- Makes it harder to diagnose production onboarding issues quickly.

8. **Modal close behavior can discard in-progress input silently**
- Clicking backdrop closes modal and clears typed state (on unmount).
- Add optional “are you sure?” if fields are non-empty.
